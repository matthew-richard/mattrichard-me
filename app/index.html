Pattern Genomics

Software Engineering Contractor

June 2014 - August 2014 (Madison, CT)

  • Programmed in C++ to reduce memory consumption and increase speed of automated assay design software. Measured tradeoffs between size and performance using profiling utilities.
  • The name of the automated assay design software was Daydreamer. We ran tests on a shared machine named "darwin" that ran CentOS Linux. It searched for DNA substrings, aka oligonucleotides aka oligos, that were unique to the genomes of various pathogens.
  • SeqAn library was used to represent DNA sequences.
  • MPI was used for communication across threads.
  • Implemented memory mapping, bit packing, and pooled allocation of DNA sequences, achieving 10% memory decrease and 20% speed increase.
    • It turned out that bit-packed sequences could take up *more* memory because in addition to the packed sequence, they stored a 64-bit integer representing the length of the sequence. For 32-nucleotide sequences (8 bytes when packed, plus 8 bytes of length int64 = 16 bytes) or shorter, they could take up as much or more space than the unpacked sequences (which used one byte per 2 nucleotides).
    • I was able to slightly mitigate this issue by storing all referenced sequences in one contiguous block, so that sequences that weren't byte-aligned no longer took up a few extra bits of memory.
    • Additional space was taken up by the memory overhead of allocating a large number of short memory segments. Each memory segment was word- or page-aligned and so took up more space than the data structures the memory was used for. This was mitigated by using a "pooled" memory strategy where pages of memory were allocated at a time and then the application managed them from there.
    • The pooled memory strategy also included a per-page stack for freed elements, so that memory for freed elements could be reused. Once the stack became the size of the page, the page itself could be deallocated. There may also have been "compacting" step (sort of like garbage collection), where once a page crossed a certain "freed" threshold, I moved elements from that page to other pages and freed the page.
    • Memory mapping is the act of mapping the contents of a file to the virtual memory of a process. This doesn't necessarily require the entire file to be loaded into memory at once, though of course it requires there to be a large enough address space to contain every location in the file (Not a problem on 64-bit systems). Typically, the file is loaded into memory one page at a time, lazily (i.e. if a portion of the file is never accessed, it is not loaded into memory), and the physical memory used is the OS's file cache / page cache. Benefits: Avoid read/write system calls, which are expensive; avoid "thrashing", which is simultaneous reading/writing from OS cache to to disk which happens when trying to load into RAM a file that's bigger than the memory available; save memory due to lazy loading; linear file access is faster, since it only requires disk access when a new page boundary is crossed.
    • Memory mapping was useful in the initial phase of Daydreamer, where it was loading entire genomes and building up lists of candidate oligos. During those phases, it was cheap to store oligos as (head,len) tuples, where head was a pointer to the beginning of the oligo in the genome string. In later phases, the genomes were unloaded, so this representation was no longer useful.
    • Reduced peak RAM (RSS - resident set size ram, i.e. ram in physical memory) from 19.9 GB to 17.2 GB. Reduced time to complete from 37 min to 23 min.
  • Collected and analyzed profiling data using Valgrind's Callgrind tool, and used KCachegrind to visualize its output.
  • Worked in a Linux-based environment and wrote Bash scripts to automate development tasks.
